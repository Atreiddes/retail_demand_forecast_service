# Сервис прогноза спроса

Берёт модель недельного прогноза спроса (LightGBM по данным M5) и оборачивает её в сервис. Планер выбирает срез ассортимента, сервис считает прогноз по всем рядам этого среза, раскидывая работу по нескольким воркерам, и показывает результат с интервалами неопределённости.

## Запуск

Нужен только Docker. Из каталога stage8:

```
docker compose up -d --build
```

Поднимется весь стек: PostgreSQL, RabbitMQ, API на FastAPI, воркер с моделью, nginx, а также Prometheus, Grafana и экспортёр Postgres для мониторинга. При пустой базе она наполняется срезом M5 (категория FOODS, 14 370 рядов товар-магазин) автоматически отдельным init-шагом. Сервис доступен по адресам:

- интерфейс планера - http://localhost
- REST и Swagger - http://localhost/api/docs
- панель RabbitMQ - http://localhost:15672 (app / app)
- дашборды Grafana - http://localhost:3000 (просмотр открыт без логина)
- Prometheus - http://localhost:9090

Если порты заняты, их можно сдвинуть переменными: `WEB_PORT=8080 RMQ_MGMT_PORT=15680 DB_PORT=5442 GRAFANA_PORT=3300 PROM_PORT=9190 docker compose up -d --build`.

## Что делает планер

На главной выбирается магазин (категория FOODS, горизонт 8 недель) и запускается прогон. Запрос дробится на пачки рядов и уходит в очередь, воркеры их разбирают. На странице прогона видно, как растёт прогресс по пачкам, по готовности доступны:

- таблица рядов с суммарным прогнозом и шириной интервала P10-P90 (узкий интервал - спрос предсказуем, широкий - стоит присмотреться);
- график выбранного ряда: история продаж, точечный прогноз и лента P10-P90 на 8 недель вперёд;
- таблица метрик качества модели (WRMSSE по уровням иерархии, бэктест walk-forward);
- выгрузка прогноза в CSV.

## Масштабирование

Ряды независимы, поэтому чем больше воркеров, тем быстрее считается прогон.

```
docker compose up -d --scale worker=4
```

Замер: крупный прогон (весь FOODS, без выбора магазина) при одном воркере и при четырёх занял около 34 и 13 секунд. Дальше упирается в запись в Postgres и в неделимую сборку метрик.

## Мониторинг

Вместе со стеком поднимаются Prometheus и Grafana. Приложение отдаёт метрики, Prometheus их собирает, Grafana рисует. Источник данных и три дашборда провижинятся автоматически при старте:

- Сервис прогноза - поставленные и активные прогоны по статусам, скорость обработки пачек и время на пачку, частота и задержка HTTP-запросов, метрики качества модели (WRMSSE) и PSI дрейфа признаков по последнему завершённому прогону.
- Очередь RabbitMQ - сообщения в очереди (готовы и в обработке), соединения, потребители, скорость публикации и доставки.
- База Postgres - соединения, транзакции, размер базы.

Метрики приложения на `/metrics` отдаёт API, воркеры поднимают свой сервер метрик на порту 9100; при `--scale worker=N` Prometheus находит все реплики через DNS сервиса. Метрики брокера даёт штатный плагин rabbitmq_prometheus, метрики Postgres - postgres-exporter.

Деградация модели. По мере прихода фактических продаж сервис считает точность прогноза против факта: WMAPE, смещение (систематический пере- или недопрогноз) и покрытие интервала P10-P90, плюс разрезы по горизонту, штату (плановый уровень), промо против базовых недель, сегментам движения и cold-start, и forecast value add против базы MA-4. Дополнительно отслеживаются здоровье данных и стабильность прогноза: свежесть и полнота последней недели факта, дрейф ассортимента (новые и выбывшие ряды), стабильность прогноза (разброс P50 на одну неделю ряда между прогонами с разным origin). Гейт деградации сводит точность, калибровку интервалов, дрейф, плановое смещение, точность в промо, проигрыш базе MA-4, неполноту данных и нестабильность прогноза в флаг с предупреждениями (`GET /api/monitoring`); пороги в `monitoring.py`. На эти сигналы заведены правила алертов Prometheus. Переобучение запускается не только по расписанию: DAG `monitor_and_retrain` (Airflow, ops-слой) периодически опрашивает гейт и при деградации запускает переобучение `retrain_artifact`, где новый артефакт ещё проходит порог качества перед выкаткой.

## Как устроено

Поток один: nginx -> FastAPI -> очередь RabbitMQ -> воркер -> Postgres.

- API принимает запрос на прогноз, делит выбранные ряды на пачки (группы по несколько сотен рядов), кладёт их в очередь и сразу отдаёт id прогона, не дожидаясь расчёта.
- Воркер это отдельный фоновый процесс с моделью. Берёт пачку из очереди, читает историю своих рядов из Postgres, строит признаки, применяет предобученную модель и пишет точки прогноза. Подтверждает сообщение только после записи, поэтому падение воркера не теряет пачку. Воркеров запускают несколько, и они разбирают очередь параллельно.
- Готовность прогона определяется по статусам пачек в базе, без отдельного оркестратора.

Используется предобученная модель с проверенным качеством: артефакт (точечная LightGBM Tweedie, две квантильные модели на P10 и P90, калибровочный фактор, словарь категорий) собран заранее скриптом `scripts/train_artifact.py`. Качество измерено по схеме walk-forward: origin (граница обучения) скользит по времени, на каждом окне модель учится на истории до origin и прогнозирует 8 недель вперёд, прогноз сравнивается с фактом. Метрика - WRMSSE по 12 уровням иерархии M5, от общего итога до отдельной пары товар-магазин (полный бэктест в stage7 идёт по 8 окнам, в таблице сервиса показан разрез по уровням на последнем). Воркеры только применяют модель, поэтому их число можно наращивать без переобучения. Прогноз строится на восемь недель вперёд от последней доступной недели.

Признаки, квантили и метрики переиспользованы из stage7 (`src/forecast_service/ml`).

## Тесты

Для тестов можно поднять стек и из каталога stage8 выполнить:

```
DB_URL=postgresql+psycopg://postgres:postgres@localhost:5432/forecast uv run pytest -q
```

Покрыт критический путь: создание прогона, разбор пачки воркером, запись и финализация, идемпотентность повторной доставки (повтор не задваивает точки), монотонность p10<=p50<=p90, коды ошибок 422 и 404. Плюс перенесённые из stage7 тесты метрик WRMSSE и тесты на отсутствие утечки из будущего в признаках.

## Данные и модель

FOODS-срез панели M5, календарь и обученный артефакт модели уже лежат в репозитории (`data/`, `artifacts/`). Все необходимые зависимости в Docker image.

При необходимости артефакт пересобирается офлайн скриптами `scripts/prepare_data.py`, `scripts/train_artifact.py`, `scripts/foods_metrics.py`. Они переиспользуют обучающий пайплайн из stage5 (`stage5/scripts`), поэтому запускаются в его окружении, для работы сервиса это не требуется.
